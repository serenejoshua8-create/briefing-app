# 0700/2100 Briefing — MVP Build Guide

This turns the browser-only prototype into a real app: a small backend on **Render**,
a database on **Supabase**, code on **GitHub**, dispatches generated by **Claude**
and independently fact-checked by **Gemini**, and an **Archive** tab that merges
your own Google Drive notes with the app's generated dispatches (with a
Save-to-Drive button).

Read this top to bottom once, then come back to each phase as you execute it.

---

## Architecture

```
┌──────────────┐         ┌───────────────────────┐         ┌─────────────┐
│   Frontend    │ ── →   │   Backend (Render)     │  ── →   │  Supabase    │
│ (static HTML) │ ← ──   │   Node + Express        │  ← ──   │  Postgres    │
└──────────────┘         └───────────────────────┘         └─────────────┘
                                  │      │      │
                                  │      │      └── → Google Drive API
                                  │      │             (read notes / save dispatch)
                                  │      └── → Gemini API (independent fact-check)
                                  └── → Claude API (generate 12 summary + 5 analysis)
```

Why this shape:
- **API keys move server-side.** The prototype called `api.anthropic.com` directly
  from the browser, which meant the key was exposed in the page. Now only your
  Render backend holds the keys.
- **Supabase replaces browser storage** so dispatches survive across devices and
  don't disappear if you clear browser data.
- **Gemini cross-checks Claude.** Claude generates the dispatch; Gemini
  independently re-searches and verifies dates/facts/relevance with its own
  Google Search grounding. A story only "passes" if neither model flags it.
- **Google Drive becomes the archive of record.** Your existing Daily Notes docs
  and every new dispatch live in the same Drive folder, and the Archive tab reads
  both.

## Phase 2 — Supabase project

1. Create a new project at supabase.com. Pick any region (Mumbai/Singapore is
   closest to IST if offered).
2. Open **SQL Editor** → paste the contents of `supabase/schema.sql` → Run.
   This creates three tables: `briefings`, `weekly_digests`, `app_config`.
3. Go to **Settings → API**. Copy:
   - `Project URL` → this is `SUPABASE_URL`
   - `service_role` key (NOT the `anon` key) → this is `SUPABASE_SERVICE_KEY`

   The service_role key bypasses Row Level Security and must **only** ever live
   in your backend's environment variables — never in the frontend, never in git.

That's it for now. This is a single-user app, so we're not setting up Supabase
Auth or RLS policies for the MVP — see "Security notes" at the end for when to
add that.

## Phase 3 — Google Cloud + Drive API

This lets the backend read your existing Daily Notes docs and save new
dispatches into the same folder.

1. Go to console.cloud.google.com → create a new project (e.g. "briefing-app").
2. **APIs & Services → Library** → search "Google Drive API" → Enable.
3. **APIs & Services → OAuth consent screen**:
   - User type: External
   - App name: anything (e.g. "Briefing Room")
   - Scopes: add `https://www.googleapis.com/auth/drive.file` and
     `https://www.googleapis.com/auth/drive.readonly`
   - Test users: add your own Google account email
   - Leave it in "Testing" mode — fine for personal use, no Google review needed.
4. **APIs & Services → Credentials → Create Credentials → OAuth client ID**:
   - Application type: Web application
   - Authorized redirect URI: `http://localhost:3000/oauth/callback` (you'll swap
     this for your Render URL later, but you need localhost first to grab a
     refresh token)
   - Save the `Client ID` and `Client secret`.
5. In Google Drive, create (or pick an existing) folder for your daily notes —
   the one containing files like `Daily_Notes_May.docx`. Open it in the browser
   and copy the folder ID from the URL:
   `https://drive.google.com/drive/folders/<THIS_PART_IS_THE_ID>`

### Getting a refresh token (one-time, run locally)

```bash
cd backend
npm install
node scripts/get-refresh-token.js
```

This opens a URL in your terminal — visit it, sign in with the Google account
that owns the Drive folder, approve access. It prints a `refresh_token` — save
it as `GOOGLE_REFRESH_TOKEN` in your `.env`. You only do this once; the backend
uses the refresh token forever after (until you revoke access in your Google
Account settings).

## Phase 4 — Backend

Everything is in `backend/server.js`. Key routes:

| Route | Method | Purpose |
|---|---|---|
| `/api/config` | GET/POST | Read/save your topic selection |
| `/api/briefing/run` | POST `{session}` | Generate + verify + save a dispatch |
| `/api/briefing/:date/:session` | GET | Fetch one stored dispatch |
| `/api/briefings` | GET | List all stored dispatches (for Archive) |
| `/api/weekly/run` | POST | Generate the weekly digest |
| `/api/drive/archive` | GET | List files in your Drive notes folder |
| `/api/drive/file/:fileId` | GET | Read a Drive doc's content as HTML |
| `/api/drive/save` | POST `{date,session}` | Push a dispatch into Drive as a Doc |

Install and run locally first:

```bash
cd backend
npm install
cp .env.example .env    # fill in every value
npm start
```

Visit `http://localhost:3000/api/config` — you should get back `{"topics":[],"extraTopics":""}`.

### How the dual verification works (Phase 5, inside server.js)

1. Claude generates the 12-summary + 5-analysis dispatch with per-point quotes
   and URLs (same structure as the prototype).
2. The backend does a lightweight **URL sanity check** on every link — rejects
   anything that isn't a real path (no bare homepages like `wsj.com/`), by doing
   a `HEAD` request and checking the path has more than just a domain.
3. Gemini receives the same 12 topics as JSON and independently re-verifies
   recency, factual accuracy, and topic relevance **using its own Google Search
   grounding** — a fully separate search, not reusing Claude's.
4. **Reconciliation is conservative**: if either model says "drop", it's
   dropped. If either supplies a "fix", the fix is applied. A story only
   survives untouched if both models pass it. This catches the case where one
   model hallucinates something the other one didn't.

## Phase 6 — Deploy backend to Render

1. Push your repo to GitHub (Phase 1).
2. On render.com: **New → Web Service** → connect your GitHub repo.
3. Root directory: `backend`
4. Build command: `npm install`
5. Start command: `node server.js`
6. Add every variable from `backend/.env.example` under **Environment**.
7. Update your Google OAuth redirect URI to include your Render URL too
   (`https://your-app.onrender.com/oauth/callback`) if you ever need to
   re-authorize.
8. Deploy. Watch the logs; hit `https://your-app.onrender.com/api/config` to confirm it's alive.

### Optional but recommended: automatic AM/PM runs

Render's free tier doesn't include Cron Jobs, but Render's paid Cron Jobs
(or a free external scheduler like **cron-job.org** or **GitHub Actions on a
schedule**) can hit your own endpoint on a timer. Add a shared-secret guard
first (see `APP_SHARED_SECRET` in `.env.example` and the check in
`server.js`), then schedule:

- `POST https://your-app.onrender.com/api/briefing/run` with `{"session":"AM"}`
  at **01:30 UTC** (07:00 IST)
- Same with `{"session":"PM"}` at **15:30 UTC** (21:00 IST)

A ready-made GitHub Actions workflow for this is in `.github/workflows/schedule.yml`.
This is what finally lets the briefing "wake itself up" — the one thing the
original browser-only artifact could never do.

## Phase 7 — Frontend

`frontend/index.html` is the same visual design as the prototype, updated to:
- Call your Render backend instead of `api.anthropic.com` directly
- Merge the Archive tab: generated dispatches (from Supabase, via the backend)
  **and** your existing Drive notes (via `/api/drive/archive`), sorted together
  by date
- Add a **Save to Drive** button on every generated dispatch, which calls
  `/api/drive/save` and creates a new Google Doc in your notes folder

Set the backend URL at the top of `frontend/index.html`:

```js
const API_BASE = "https://your-app.onrender.com";
```

Deploy the frontend anywhere static: a second Render **Static Site** (root
directory `frontend`), GitHub Pages, or just open the file locally while
testing — since it's a single HTML file with no build step.

## Phase 8 — Testing checklist

- [ ] `POST /api/briefing/run {"session":"AM"}` returns 12 topics, 5 flagged `detailed:true`
- [ ] Every point's `url` is a full article path, not a homepage (spot check a few)
- [ ] Verification metadata shows on the dispatch (`fixed`/`dropped`/`clean`)
- [ ] At least 3 of 12 points cite a paywalled outlet
- [ ] Archive tab shows both Drive `.docx` files and generated dispatches, merged by date
- [ ] Clicking a Drive file in Archive renders its content (converted from docx)
- [ ] "Save to Drive" on a dispatch creates a new Doc visible in your Drive folder
- [ ] Switching Summary/Analysis tabs re-renders instantly, no new API call
- [ ] Weekly digest runs and reads from Supabase-stored dispatches

## Security notes (read before you rely on this daily)

- This MVP has **no login** — anyone with your Render URL can trigger runs and
  spend your API credits. For personal use behind a private URL this is a
  manageable risk, but at minimum:
  - Set `APP_SHARED_SECRET` in `.env` and require it as a header
    (`x-app-secret`) on every mutating route — the scaffold already checks for
    this in `server.js`, you just need to set the value.
  - Consider Render's IP allowlisting or putting the service behind Cloudflare
    Access if you want it locked to just you.
- Rotate the Anthropic/Gemini keys if this repo or `.env` is ever exposed.
- The Google refresh token grants ongoing Drive access — revoke it any time at
  myaccount.google.com/permissions if you stop using the app.
- If you ever add other users, move to real Supabase Auth + Row Level Security
  instead of the shared-secret approach.
